# 用 Cursor 做 AI 法务项目，我们是怎么把效率和质量拉上去的

> **文档编号：** Laby-DEV-CURSOR-001  
> **版本：** 2.0 · 2026-07-08  
> **项目：** [laby-ai-legal](https://github.com/laby-umr/laby-ai-legal) — AI 法务合同审核与管理平台

---

## 写在前面

laby-ai-legal 是一个典型的「大项目 + AI 辅助开发」场景：后端 Java、前端 Vue、还有 Agent、知识库、OnlyOffice 审阅等多条业务线。团队用 Cursor Agent 写代码一段时间后，有一个很直观的感受——

**Agent 能写，但不好管。**

有时一行改动很顺，有时来来回回改三四轮；有时 Review 还在纠结 import 写法，有时差点把本地数据库脚本推上 GitHub。我们做的调整，不是堆更多配置，而是回答三个问题：

1. **怎么少花 Token、少走弯路？**
2. **怎么少出 Bug、少踩坑？**
3. **怎么让代码质量稳定在一个水准上，而不是看 Agent 当天「发挥」？**

下面这篇文章，**重点讲我们调整之后的效果评估**，以及背后做了哪些事。技术细节点到为止，想深入配置的同事可以看 [.cursor/README.md](../../.cursor/README.md)。

---

## 一、调整前：我们遇到的四类问题

| 问题 | 日常表现 |
|------|----------|
| **Token 花在了不该花的地方** | Agent 经常扫到依赖包、编译产物、大文件，检索一圈下来上下文又臭又长 |
| **同样的问题反复出现** | 类名写法不规范、SQL 中文乱码、分层乱改——Review 像复读机 |
| **流程靠自觉** | 有时先写 Plan 再动手，有时直接开写，交付质量波动大 |
| **测了单测，上线前还漏项** | 接口通了，页面没点、Postman 没更、关键流程没跑一遍 |

还有一个现实教训：公开推 GitHub 时，历史 SQL 里的示例密钥触发了平台拦截。**安全和规范不能靠「记得别提交」，得靠机制。**

---

## 二、我们做了什么（用人话讲）

整体思路可以概括成四句话：

> **该看的规范，写进仓库，别每次靠聊天提醒。**  
> **该拦的错误，提交前就挡住，别等 Review 或 CI 才发现。**  
> **大改动先想清楚再写，别边写边推翻。**  
> **写完要过一套检查清单，别「单测绿了就算完」。**

具体落地，大致是这些方向（不展开文件清单）：

| 方向 | 做了什么 | 解决什么 |
|------|----------|----------|
| **减噪音** | 让 Cursor 别索引依赖、编译目录、本地 SQL 等 | Token 少浪费、少误改 |
| **分角色** | 改法务、改 AI、改前端，各走各的关注点 | 少灌无关上下文 |
| **定流程** | 新功能先 Spec/Plan，修 Bug 先找根因 | 少返工、少猜修 |
| **设门禁** | 提交前检查常见问题；禁止误提交 SQL、强推主分支 | 少低级事故 |
| **验交付** | 单测 → 接口冒烟 → 页面走一遍 → API 文档同步 | 少「以为好了」 |

另外接入了架构图、接口测试、浏览器点页面等工具，但定位是**辅助验证**，不是替代人和 Git 上的主文档。

## 三、核心：调整后的提升评估

> 说明：以下是结合机制推导的**区间估算**，不是严格 A/B 实验数据。团队若连续用 2～4 周，可以用「每任务对话轮数、Review 打回次数、冒烟通过率」自行校准。

### 3.1 Token 消耗：大约能省多少？

| 场景 | 预估变化 | 主要原因 |
|------|----------|----------|
| 小改（1～2 个文件） | 基本持平，或略增 **5%～10%** | 开头多读一点规范，有固定开销 |
| 单模块功能（几个文件） | 净降 **15%～30%** | 少搜垃圾目录、少走错模块 |
| 跨模块 / 新功能 | 净降 **30%～50%** | 先 Plan 再写，少整段重写 |
| 纯读代码、问问题 | 降 **20%～40%** | 索引范围收窄 |

**一句话：** Token 的大头不是「多写了几行字」，而是**无效探索**——在错误目录里 grep、改完又推翻。我们的调整，主要砍的是这部分。

**盈亏平衡：** 大约 **第 3～5 个**有实质改动的任务之后，开始明显省 Token；**第 10 个任务左右**感受会更稳定。

---

### 3.2 Bug 率：哪些 Bug 明显少了？

| Bug 类型 | 预估下降 | 靠什么挡 |
|----------|----------|----------|
| 类名/import 写法乱 | **80%～95%** | 规范 + 提交前自动检查 |
| SQL 乱码、缺注释、字符集不对 | **70%～90%** | 写 SQL 的硬性约定 |
| 分层乱改、魔法数字、吞异常 | **50%～70%** | 全局编码约定 + 业务角色指引 |
| API 改了，文档/Collection 没跟 | **60%～80%** | 交付清单里强制同步 |
| 前端页面回归漏测 | **40%～60%** | 主流程页面走查 |
| 密钥、SQL 误提交 Git | **接近归零** | 提交拦截 + 公开库不含 SQL |

**综合看：**

- **Agent 容易犯、人 Review 也烦的那类问题**：大约少 **四成到五成**
- **合并后一周内发现的缺陷**：大约少 **四分之一到四成**（单测 + 冒烟 + 走查叠在一起）
- **严重安全事故**（密钥进仓库等）：这类靠机制硬拦，不应再靠运气

**仍要人把关的：** 业务理解偏了、边界条件漏了、和 OnlyOffice 这类第三方集成的偶发问题——规范能减蠢错，替不了产品判断和完整验收。

---

### 3.3 代码质量：Review 轻松了多少？

| 质量维度 | 预估改善 | 体感 |
|----------|----------|------|
| 风格、规范类打回 | **少一半到七成** | PR 里少吵 import、SQL 格式 |
| 模块边界、耦合 | **少三成到四成** | 改法务时不乱动 AI 模块 |
| PR 可读性 | **明显提升** | 鼓励小步改动、少无关文件 |
| 新人上手 | **快三成到五成** | 规范在仓库里，不用口口相传 |
| 设计和实现跑偏 | **少四成到六成** | 大改先 Plan，架构图有地方放 |

如果用「一个 PR 平均几轮 Review 才过」来衡量，**少 0.5～1.5 轮**是合理预期——尤其是规范类 comment 会大幅减少。

---

### 3.4 一张总表（给老板/同事看）

| 维度 | 调整前（体感） | 调整后（预估） |
|------|----------------|----------------|
| **Token / 任务** | 波动大，大功能常返工 | 单模块 ↓15%～30%，大功能 ↓30%～50% |
| **可预防 Bug** | 同类问题反复出现 | ↓40%～55% |
| **Review 规范项** | 几乎每 PR 都有 | ↓50%～70% |
| **交付漏项**（接口/页面/文档） | 时有发生 | 清单化后明显减少 |
| **安全误提交** | 靠人工记忆 | 机制拦截，风险极低 |

## 四、为什么有效：四个杠杆，不是某一个「神配置」

很多人问：「到底是哪一条 Rule 或 Skill 起作用？」

我们的体会是，**单独任何一条都不够**，叠在一起才有复利：

```
① 少索引垃圾目录     →  少花 Token、少误改
② 按模块/load 角色   →  上下文更准
③ 大改先 Plan 再写   →  少推翻重来
④ 交付前跑检查清单   →  少「以为完成了」
```

可以把它想成给 Agent 配了：**地图（项目结构）+ 交规（编码约定）+ 考试（提交前检查）+ 路考（冒烟和走查）**。缺任何一环，效果都会打折。

---

## 五、投入与代价（实话实说）

| 项目 | 说明 |
|------|------|
| **一次性成本** | 搭规范、流程、门禁、文档，大约几小时到一两天 |
| **每个任务的额外成本** | 小改可能多读 1～2 份说明；大功能要多写 Spec/Plan |
| **维护成本** | 业务变了，规范要跟着更新，否则会过时 |
| **执行率** | 跳过 Plan、不做冒烟、不跑单测 → 提升可能只剩一成左右 |

**不适合的场景：** 改一行注释、问一个概念——走全套流程反而啰嗦。规范里也写了：**小任务不必硬套大流程。**

---

## 六、团队怎么用（三步就够）

1. **接需求时先分类型**  
   - 新功能 / 大改 → 先想清楚再写（Spec/Plan）  
   - 修 Bug → 先找根因，禁止「猜一下改改看」  
   - 小改动 → 直接改，但编码约定仍要遵守  

2. **写完别急着说「好了」**  
   - 单测过  
   - 关键接口冒烟过  
   - 动过页面的，主流程点一遍  
   - 动过 API 的，同步接口文档  

3. **提交前心里过一遍**  
   - 有没有混进本地 SQL、密钥、无关文件  
   - 改的是不是「最小必要范围」  

更细的入口见 [.cursor/README.md](../../.cursor/README.md)，本文不展开。

## 七、和 GitHub 公开仓库怎么配合

我们把 [laby-ai-legal](https://github.com/laby-umr/laby-ai-legal) 作为对外展示仓库时，策略和上面一致：

- **公开：** 源码、Cursor 协作配置、交付文档、接口 Collection、Docker 编排  
- **不公开：** 本地 SQL 初始化脚本（含环境信息与历史示例密钥）  
- **README 写清楚：** 克隆后数据库脚本怎么获取，避免新人踩空  

本地「不能提交 SQL」和远程「仓库里本来就没有 SQL」是同一套安全思路的两面。

## 八、后续怎么验证这些数字

如果希望把「区间估算」变成「我们自己的数据」，建议连续记录 2～4 周：

| 记什么 | 怎么记 | 目标参考 |
|--------|--------|----------|
| 每任务对话轮数 / Token | Cursor 用量或主观感受 | 比调整前 ↓20% 左右 |
| 同一需求返工几次 | 任务结束时记一笔 | 少 1 轮 |
| Review 规范类 comment | PR 里数一下 | ↓50% 以上 |
| 冒烟 / 走查通过率 | 脚本或清单勾选 | ≥95% |
| 误提交 SQL / 密钥 | 应为 0 | 0 |

有 baseline 之后，这篇文章里的区间可以换成你们自己的结论。

## 九、总结

对 laby-ai-legal 这种 **多模块 + AI + 法务合规** 的项目，用 Cursor 开发，瓶颈往往不在「模型会不会写代码」，而在：

- **上下文干不干净**  
- **流程稳不稳定**  
- **错误拦不拦得住**  
- **交付验没验完整**  

我们这套调整，**核心价值不是配置本身**，而是把协作经验固化进仓库，让 Agent 的发挥从「看当天状态」变成「大体可预期」。

**用数字概括（再次强调为估算）：**

- Token：常见任务 **省约两成**，大功能 **省约三成到五成**  
- Bug：可预防的那类 **少约四成到五成**  
- 质量：Review 规范项 **少约一半到七成**  

如果你也在做类似体量的 AI 辅助开发项目，不妨先拿「减噪音 + 先 Plan + 交付清单 + 提交门禁」这四件事试一轮——通常比继续堆对话技巧更划算。
